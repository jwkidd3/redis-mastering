Cache-aside Pattern

Most Common Pattern: Application manages both cache and database

How It Works

  1. Read: Check cache first, fallback to database
  2. Write: Update database, then invalidate cache
  3. Miss: Load from database, populate cache

JavaScript Implementation


async function getCustomerProfile(customerId) {
    const cacheKey = `customer:${customerId}`;
    
    // Try cache first
    let customer = await redis.get(cacheKey);
    if (customer) {
        return JSON.parse(customer);
    }
    
    // Cache miss - load from database
    customer = await database.getCustomer(customerId);
    
    // Populate cache with TTL
    await redis.setex(cacheKey, 3600, JSON.stringify(customer));
    
    return customer;
}

Cache-aside: Benefits & Trade-offs

✅ Benefits

  • Simple to implement
  • Cache failures don't break application
  • Data consistency guarantees
  • Flexible cache policies
  • Works with any database

⚠️ Trade-offs

  • Cache miss penalty
  • Potential data staleness
  • Complex invalidation logic
  • Race conditions possible
  • Application complexity
Pro Tip: Use cache-aside for read-heavy workloads with acceptable staleness tolerance

Write-through Caching

Synchronous Pattern: Cache and database updated together

Implementation Strategy


async function updateCustomerProfile(customerId, profileData) {
    const cacheKey = `customer:${customerId}`;
    
    try {
        // Update database first
        await database.updateCustomer(customerId, profileData);
        
        // Update cache immediately
        await redis.setex(cacheKey, 3600, JSON.stringify(profileData));
        
        return { success: true };
    } catch (error) {
        // Rollback on failure
        await redis.del(cacheKey);
        throw error;
    }
}

When to Use

  • Strong consistency required
  • Read-heavy workloads
  • Critical data updates

Considerations

  • Higher write latency
  • Cache always current
  • Complex error handling

Write-behind (Write-back) Caching

Asynchronous Pattern: Cache updated immediately, database later

Implementation Approach


async function updateCustomerQuick(customerId, updates) {
    const cacheKey = `customer:${customerId}`;
    
    // Update cache immediately
    await redis.setex(cacheKey, 3600, JSON.stringify(updates));
    
    // Queue for database update
    await redis.lpush('db_update_queue', JSON.stringify({
        table: 'customers',
        id: customerId,
        data: updates,
        timestamp: Date.now()
    }));
    
    return { success: true, cached: true };
}

// Background worker
async function processDatabaseUpdates() {
    const update = await redis.brpop('db_update_queue', 0);
    const updateData = JSON.parse(update[1]);
    
    await database.updateCustomer(updateData.id, updateData.data);
}

Cache Invalidation Strategies

Remember: "There are only two hard things in Computer Science: cache invalidation and naming things" - Phil Karlton

Time-based Invalidation (TTL)


// Set with expiration
await redis.setex('policy:12345', 3600, policyData);

// Check remaining time
const ttl = await redis.ttl('policy:12345');

// Refresh before expiration
if (ttl < 300) { // Less than 5 minutes
    await refreshPolicyCache('12345');
}

Event-based Invalidation


async function updatePolicy(policyId, newData) {
    // Update database
    await database.updatePolicy(policyId, newData);
    
    // Invalidate related caches
    await Promise.all([
        redis.del(`policy:${policyId}`),
        redis.del(`customer:policies:${newData.customerId}`),
        redis.del(`agent:policies:${newData.agentId}`)
    ]);
}

Tag-based Cache Invalidation

Grouping Related Cache Keys


async function cacheWithTags(key, data, ttl, tags) {
    // Store the data
    await redis.setex(key, ttl, JSON.stringify(data));
    
    // Associate with tags
    for (const tag of tags) {
        await redis.sadd(`tag:${tag}`, key);
        await redis.expire(`tag:${tag}`, ttl + 300); // Slightly longer TTL
    }
}

// Usage example
await cacheWithTags(
    'policy:12345',
    policyData,
    3600,
    ['customer:1001', 'agent:205', 'policy_type:auto']
);

// Invalidate by tag
async function invalidateByTag(tag) {
    const keys = await redis.smembers(`tag:${tag}`);
    
    if (keys.length > 0) {
        await redis.del(...keys);
        await redis.del(`tag:${tag}`);
    }
}

Multi-level Caching Architecture

L1: Application Cache

  • In-memory objects
  • Fastest access
  • Limited capacity
  • Process-local

L2: Redis Cache

  • Shared across instances
  • Network overhead
  • Larger capacity
  • Persistent storage

L3: Database

  • Source of truth
  • Highest latency
  • Full dataset
  • ACID guarantees

Implementation Pattern


class MultiLevelCache {
    constructor() {
        this.l1Cache = new Map(); // In-memory
        this.redis = redis;       // L2 cache
    }
    
    async get(key) {
        // L1 check
        if (this.l1Cache.has(key)) {
            return this.l1Cache.get(key);
        }
        
        // L2 check
        const l2Data = await this.redis.get(key);
        if (l2Data) {
            const parsed = JSON.parse(l2Data);
            this.l1Cache.set(key, parsed); // Populate L1
            return parsed;
        }
        
        // L3 fallback (database)
        return null;
    }
}

Pattern Consistency Performance Complexity Best For
Cache-aside Eventually Good Medium General purpose

Cache Performance Optimization

Key Strategies

Reduce Network Calls


// Batch operations
const pipeline = redis.pipeline();
pipeline.get('policy:1');
pipeline.get('policy:2');
pipeline.get('policy:3');
const results = await pipeline.exec();

// Multi-get
const values = await redis.mget(
    'policy:1', 'policy:2', 'policy:3'
);

Optimize Data Structure


// Use hashes for partial updates
await redis.hset('policy:12345', {
    'status': 'active',
    'premium': '1200',
    'last_updated': Date.now()
});

// Get specific fields only
const status = await redis.hget(
    'policy:12345', 'status'
);

Memory Optimization

  • Compression: Use shorter keys and compress large values
  • TTL Management: Set appropriate expiration times
  • Data Structure Choice: Use most efficient Redis type
  • Eviction Policies: Configure memory limits and eviction

Error Handling & Resilience

Circuit Breaker Pattern


class CacheCircuitBreaker {
    constructor(threshold = 5, timeout = 60000) {
        this.failureCount = 0;
        this.threshold = threshold;
        this.timeout = timeout;
        this.state = 'CLOSED';
        this.nextAttempt = Date.now();
    }
    
    async execute(cacheOperation, fallback) {
        if (this.state === 'OPEN') {
            if (Date.now() < this.nextAttempt) {
                return await fallback();
            }
            this.state = 'HALF_OPEN';
        }
        
        try {
            const result = await cacheOperation();
            this.onSuccess();
            return result;
        } catch (error) {
            this.onFailure();
            return await fallback();
        }
    }
    
    onSuccess() {
        this.failureCount = 0;
        this.state = 'CLOSED';
    }
    
    onFailure() {
        this.failureCount++;
        if (this.failureCount >= this.threshold) {
            this.state = 'OPEN';
            this.nextAttempt = Date.now() + this.timeout;
        }
    }
}

Cache Monitoring & Metrics

Key Metrics to Track

Performance Metrics

  • Cache hit ratio
  • Average response time
  • Cache miss penalty
  • Throughput (ops/sec)

Health Metrics

  • Memory usage
  • Eviction rate
  • Connection pool status
  • Error rates

Implementation


class CacheMetrics {
    constructor() {
        this.hits = 0;
        this.misses = 0;
        this.errors = 0;
    }
    
    recordHit() { this.hits++; }
    recordMiss() { this.misses++; }
    recordError() { this.errors++; }
    
    getHitRatio() {
        const total = this.hits + this.misses;
        return total > 0 ? this.hits / total : 0;
    }
    
    getStats() {
        return {
            hitRatio: this.getHitRatio(),
            totalOps: this.hits + this.misses,
            errorRate: this.errors / (this.hits + this.misses + this.errors)
        };
    }
}

Caching Best Practices

🎯 Design

  • Cache at multiple levels
  • Use appropriate TTLs
  • Design for cache failures
  • Monitor cache performance

🚀 Performance

  • Batch operations when possible
  • Use pipelines for multiple commands
  • Optimize key naming
  • Choose right data structures

🛡️ Reliability

  • Implement circuit breakers
  • Handle cache stampedes
  • Plan for cache warming
  • Test failure scenarios
Anti-patterns to Avoid:
  • Caching everything without strategy
  • No TTL or inappropriate TTL values
  • Ignoring cache invalidation
  • Not monitoring cache performance
  • Blocking application on cache failures
