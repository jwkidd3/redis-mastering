Redis Hashes Overview

What are Hashes?

  • Field-value pairs within a single key
  • Perfect for object-like structures
  • Memory efficient for small to medium objects
  • Support partial updates and atomic operations

Insurance Applications

  • Customer Profiles: Name, contact, preferences
  • Policy Records: Coverage, premium, status
  • Claims Data: Amount, status, dates
  • Agent Information: Territory, performance, contact
Key Benefit: Update individual fields without affecting the entire record, perfect for real-time customer portal updates and policy modifications.

Basic Hash Operations

Command Description Time Complexity
HSET key field value Set field in hash O(1)
HGET key field Get field value O(1)
HGETALL key Get all fields and values O(N)
HMSET key field1 value1 field2 value2 Set multiple fields O(N)
HMGET key field1 field2 Get multiple fields O(N)
HDEL key field Delete field O(1)
HEXISTS key field Check if field exists O(1)

Customer Profile with Hashes

# Create comprehensive customer profile
HSET customer:12345 name "Alice Johnson"
HSET customer:12345 email "alice.johnson@example.com"
HSET customer:12345 phone "555-0123"
HSET customer:12345 tier "premium"
HSET customer:12345 join_date "2023-01-15"
HSET customer:12345 risk_score "850"
HSET customer:12345 state "CA"
HSET customer:12345 preferred_contact "email"

# More efficient: Set multiple fields at once
HMSET customer:67890 name "Bob Smith" email "bob@example.com" \
    phone "555-0456" tier "standard" join_date "2023-02-20" \
    risk_score "720" state "TX" preferred_contact "phone"

# Retrieve specific customer information
HGET customer:12345 name          # Returns: "Alice Johnson"
HGET customer:12345 tier          # Returns: "premium"

# Get multiple fields efficiently
HMGET customer:12345 name email tier  # Returns: ["Alice Johnson", "alice.johnson@example.com", "premium"]

# Get all customer data
HGETALL customer:12345  # Returns all field-value pairs

Policy Management with Hashes

# Auto insurance policy
HMSET policy:AUTO-001 customer_id "12345" type "auto" \
    premium "1200.50" deductible "500" status "active" \
    coverage_limit "100000" vehicle "2022 Honda Civic" \
    start_date "2024-01-01" renewal_date "2025-01-01"

HMSET policy:HOME-001 customer_id "12345" type "home" \
    premium "850.00" deductible "1000" status "active" \

HSET policy:AUTO-001 premium "1150.00"

# Check if customer has specific coverage
HEXISTS policy:AUTO-001 comprehensive_coverage

# Get policy summary for customer portal
HMGET policy:AUTO-001 type premium status renewal_date

Hashes vs JSON Strings for Insurance Data

üèÜ Redis Hashes

  • Partial Updates: Change single fields
  • Memory Efficient: Optimized encoding
  • Field-Level Operations: HINCRBY, HEXISTS
  • Type Safety: Field-based validation
  • Atomic Operations: Per-field guarantees

üìã JSON Strings

  • Complex Nesting: Deep object structures
  • Array Support: Lists within objects
  • Schema Flexibility: Dynamic structures
  • Full Replacement: All-or-nothing updates
  • Standard Format: API compatibility
Best Practice: Use Hashes for frequently updated fields (customer contact info, policy status). Use JSON for complex nested data (claim details with multiple adjusters, coverage definitions).

Advanced Hash Operations

Numeric Operations

  • HINCRBY - Increment integer field
  • HINCRBYFLOAT - Increment float field
  • HLEN - Get number of fields

Field Management

  • HKEYS - Get all field names
  • HVALS - Get all values
  • HSETNX - Set if field doesn't exist
# Increment claim count
HINCRBY customer:12345 claim_count 1

# Update premium with float precision
HINCRBYFLOAT policy:AUTO-001 premium 25.50

# Conditional field setting
HSETNX customer:12345 signup_bonus "100"

# Get field count
HLEN policy:AUTO-001  # Returns number of fields

# Get all field names
HKEYS policy:AUTO-001

Redis Lists Overview

What are Lists?

  • Ordered sequences of strings
  • Support both stack and queue operations
  • Fast insertion at head and tail
  • Perfect for workflow processing

Insurance Applications

  • Claims Processing Queue: FIFO order
  • Policy Renewal Reminders: Scheduled tasks
  • Audit Logs: Chronological events
  • Notification Queue: Customer alerts
Key Feature: Blocking operations allow workers to wait for new tasks, enabling efficient real-time processing of claims and policy updates.

Basic List Operations

Command Description Use Case
LPUSH key value Add to list head (left) Stack, recent items first
RPUSH key value Add to list tail (right) Queue, FIFO processing
LPOP key Remove from head Process most recent
RPOP key Remove from tail Process oldest in queue
LLEN key Get list length Queue size monitoring
LRANGE key start stop Get range of elements View queue contents
BLPOP key timeout Blocking pop from head Real-time processing

Claims Processing Queue with Lists

# Add new claims to processing queue (FIFO)
RPUSH claims:processing "CLAIM-2024-001:fender_bender:2500"
RPUSH claims:processing "CLAIM-2024-002:water_damage:8500"
RPUSH claims:processing "CLAIM-2024-003:theft:15000"

# Add urgent claim to front of queue
LPUSH claims:urgent "CLAIM-2024-004:total_loss:35000"

# Check queue sizes
LLEN claims:processing  # Returns: 3
LLEN claims:urgent      # Returns: 1

# View pending claims (don't remove)
LRANGE claims:processing 0 -1  # Show all pending claims
LRANGE claims:urgent 0 4       # Show next 5 urgent claims

# Process next claim (remove from queue)
LPOP claims:urgent           # Process urgent first
RPOP claims:processing       # Process oldest regular claim

# Worker process with blocking (waits for new claims)
BLPOP claims:urgent claims:processing 30  # Wait 30 seconds for work

const redis = require('redis');
const client = redis.createClient();

// Producer: Add claims to processing queue
async function submitClaim(claimData) {
    const claimId = `CLAIM-${Date.now()}`;
    await client.rPush('claims:processing', JSON.stringify({
        id: claimId,
        ...claimData,
        timestamp: new Date().toISOString()
    }));
    console.log(`Claim ${claimId} submitted for processing`);
}

// Consumer: Process claims with blocking wait
async function processClaimsWorker() {
    while (true) {
        try {
            // Wait up to 30 seconds for new claim
            const result = await client.blPop('claims:processing', 30);
            if (result) {
                const claimData = JSON.parse(result.element);
                await processClaim(claimData);
            }
        } catch (error) {
            console.error('Error processing claims:', error);
        }
    }
}

Memory Optimization for Insurance Data

Hash Optimization

  • Small Hashes: Use zip encoding for efficiency
  • Field Names: Keep short but meaningful
  • Expiration: Set TTL for temporary data
  • Compression: Consider for large text fields

List Optimization

  • Trim Lists: Use LTRIM to limit size
  • Batch Operations: Multiple RPUSH in one call
  • Queue Monitoring: Track length for alerts
  • Dead Letter Queues: Handle failed processing
# Keep recent customer activity (last 100 actions)
LPUSH customer:12345:activity "viewed_policy"
LTRIM customer:12345:activity 0 99

# Batch claim submission
RPUSH claims:processing "CLAIM-001" "CLAIM-002" "CLAIM-003"

# Monitor queue depth
LLEN claims:processing

Performance Best Practices

Hash Performance

  • HGETALL: Avoid on large hashes
  • Partial Updates: Use HSET for single fields
  • Batch Operations: HMGET/HMSET for multiple fields
  • Field Indexing: Consider separate indexes

List Performance

  • Head/Tail Operations: O(1) performance
  • Middle Operations: Avoid LINSERT, LSET
  • Range Queries: Limit LRANGE size
  • Blocking Operations: Use reasonable timeouts
