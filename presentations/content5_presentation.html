Redis Hashes Overview

What are Hashes?

  • Field-value pairs within a single key
  • Perfect for object-like structures
  • Memory efficient for small to medium objects
  • Support partial updates and atomic operations

Insurance Applications

  • Customer Profiles: Name, contact, preferences
  • Policy Records: Coverage, premium, status
  • Claims Data: Amount, status, dates
  • Agent Information: Territory, performance, contact
Key Benefit: Update individual fields without affecting the entire record, perfect for real-time customer portal updates and policy modifications.

Basic Hash Operations

Command Description Time Complexity
HSET key field value Set field in hash O(1)
HGET key field Get field value O(1)
HGETALL key Get all fields and values O(N)
HMSET key field1 value1 field2 value2 Set multiple fields O(N)
HMGET key field1 field2 Get multiple fields O(N)
HDEL key field Delete field O(1)
HEXISTS key field Check if field exists O(1)

Customer Profile with Hashes

# Create comprehensive customer profile
HSET customer:12345 name "Alice Johnson"
HSET customer:12345 email "alice.johnson@example.com"
HSET customer:12345 phone "555-0123"
HSET customer:12345 tier "premium"
HSET customer:12345 join_date "2023-01-15"
HSET customer:12345 risk_score "850"
HSET customer:12345 state "CA"
HSET customer:12345 preferred_contact "email"

# More efficient: Set multiple fields at once
HMSET customer:67890 name "Bob Smith" email "bob@example.com" \
    phone "555-0456" tier "standard" join_date "2023-02-20" \
    risk_score "720" state "TX" preferred_contact "phone"

# Retrieve specific customer information
HGET customer:12345 name          # Returns: "Alice Johnson"
HGET customer:12345 tier          # Returns: "premium"

# Get multiple fields efficiently
HMGET customer:12345 name email tier  # Returns: ["Alice Johnson", "alice.johnson@example.com", "premium"]

# Get all customer data
HGETALL customer:12345  # Returns all field-value pairs

Hashes vs JSON Strings for Insurance Data

üèÜ Redis Hashes

  • Memory Efficient: Optimized encoding
  • Field-Level Operations: HINCRBY, HEXISTS
  • Type Safety: Field-based validation
  • Atomic Operations: Per-field guarantees

üìã JSON Strings

  • Complex Nesting: Deep object structures
  • Array Support: Lists within objects
  • Schema Flexibility: Dynamic structures
  • Full Replacement: All-or-nothing updates
  • Standard Format: API compatibility
Best Practice: Use Hashes for frequently updated fields (customer contact info, policy status). Use JSON for complex nested data (claim details with multiple adjusters, coverage definitions).

Advanced Hash Operations

Numeric Operations

  • HINCRBY - Increment integer field
  • HINCRBYFLOAT - Increment float field
  • HLEN - Get number of fields

Field Management

  • HKEYS - Get all field names
  • HVALS - Get all values
  • HSETNX - Set if field doesn't exist
# Increment claim count
HINCRBY customer:12345 claim_count 1

# Update premium with float precision

# Get field count
HLEN policy:AUTO-001  # Returns number of fields

# Get all field names
HKEYS policy:AUTO-001

Redis Lists Overview

What are Lists?

  • Ordered sequences of strings
  • Support both stack and queue operations
  • Fast insertion at head and tail
  • Perfect for workflow processing

Insurance Applications

  • Claims Processing Queue: FIFO order
  • Policy Renewal Reminders: Scheduled tasks
  • Audit Logs: Chronological events
  • Notification Queue: Customer alerts
Key Feature: Blocking operations allow workers to wait for new tasks, enabling efficient real-time processing of claims and policy updates.

Basic List Operations

Command Description Use Case
LPUSH key value Add to list head (left) Stack, recent items first
RPUSH key value Add to list tail (right) Queue, FIFO processing
LPOP key Remove from head Process most recent
RPOP key Remove from tail Process oldest in queue
LLEN key Get list length Queue size monitoring
LRANGE key start stop Get range of elements View queue contents
BLPOP key timeout Blocking pop from head Real-time processing

Claims Processing Queue with Lists

# Add new claims to processing queue (FIFO)
RPUSH claims:processing "CLAIM-2024-001:fender_bender:2500"
RPUSH claims:processing "CLAIM-2024-002:water_damage:8500"
RPUSH claims:processing "CLAIM-2024-003:theft:15000"

# Add urgent claim to front of queue
LPUSH claims:urgent "CLAIM-2024-004:total_loss:35000"

# Check queue sizes
LLEN claims:processing  # Returns: 3
LLEN claims:urgent      # Returns: 1

# View pending claims (don't remove)
LRANGE claims:processing 0 -1  # Show all pending claims
LRANGE claims:urgent 0 4       # Show next 5 urgent claims

# Process next claim (remove from queue)
LPOP claims:urgent           # Process urgent first
RPOP claims:processing       # Process oldest regular claim

# Worker process with blocking (waits for new claims)
BLPOP claims:urgent claims:processing 30  # Wait 30 seconds for work

Insurance Workflow Patterns

# Notification queue

Producer-Consumer with Blocking Operations

// JavaScript example for Lab 7 preparation
const redis = require('redis');
const client = redis.createClient();

// Producer: Add claims to processing queue
async function submitClaim(claimData) {
    const claimId = `CLAIM-${Date.now()}`;
    await client.rPush('claims:processing', JSON.stringify({
        id: claimId,
        ...claimData,
        timestamp: new Date().toISOString()
    }));
    console.log(`Claim ${claimId} submitted for processing`);
}

// Consumer: Process claims with blocking wait
async function processClaimsWorker() {
    while (true) {
        try {
            // Wait up to 30 seconds for new claim
            const result = await client.blPop('claims:processing', 30);
            if (result) {
                const claimData = JSON.parse(result.element);
                await processClaim(claimData);
            }
        } catch (error) {
            console.error('Error processing claims:', error);
        }
    }
}

Memory Optimization for Insurance Data

Hash Optimization

  • Small Hashes: Use zip encoding for efficiency
  • Field Names: Keep short but meaningful
  • Expiration: Set TTL for temporary data
  • Compression: Consider for large text fields

List Optimization

  • Trim Lists: Use LTRIM to limit size
  • Batch Operations: Multiple RPUSH in one call
  • Queue Monitoring: Track length for alerts
  • Dead Letter Queues: Handle failed processing
# Keep recent customer activity (last 100 actions)
LPUSH customer:12345:activity "viewed_policy"
LTRIM customer:12345:activity 0 99

# Batch claim submission
RPUSH claims:processing "CLAIM-001" "CLAIM-002" "CLAIM-003"

# Monitor queue depth
LLEN claims:processing

Performance Best Practices

Hash Performance

  • HGETALL: Avoid on large hashes
  • Partial Updates: Use HSET for single fields
  • Batch Operations: HMGET/HMSET for multiple fields
  • Field Indexing: Consider separate indexes

List Performance

  • Head/Tail Operations: O(1) performance
  • Middle Operations: Avoid LINSERT, LSET
  • Range Queries: Limit LRANGE size
  • Blocking Operations: Use reasonable timeouts

Redis Sets - Unordered Collections

What are Sets?

  • Unordered collection of unique strings
  • O(1) add, remove, and membership testing
  • Support set algebra operations
  • Perfect for relationships and tags

Insurance Applications

  • Customer Tags: Risk categories, preferences
  • Active Claims: Unique claimant tracking
  • Agent Skills: Certifications, specializations
  • Policy Features: Coverage types, riders
Key Feature: Automatic uniqueness guarantee - no duplicate members. Perfect for tracking distinct entities like active customers, unique claimants, or policy holders.

Basic Set Operations

Command Description Time Complexity
SADD key member [member ...] Add one or more members to set O(N) where N is number of members
SISMEMBER key member Check if member exists in set O(1)
SMEMBERS key Get all members of set O(N)
SCARD key Get number of members (cardinality) O(1)
SREM key member [member ...] Remove one or more members O(N)
Use Case: Track active claimants, unique policy holders per state, or customers with specific coverage types.

Sets for Customer Analytics

# Track customers by risk category
SADD customers:high_risk "customer:12345" "customer:67890" "customer:11111"
SADD customers:low_risk "customer:22222" "customer:33333" "customer:44444"

# Check if customer is in high risk category
SISMEMBER customers:high_risk "customer:12345"  # Returns 1 (true)

# Get all high-risk customers
SMEMBERS customers:high_risk
# Returns: ["customer:12345", "customer:67890", "customer:11111"]

# Count high-risk customers
SCARD customers:high_risk  # Returns: 3

# Customer moved to low risk
SREM customers:high_risk "customer:12345"
SADD customers:low_risk "customer:12345"

# Track active claimants (unique only)
SADD claimants:active "claimant:1001" "claimant:1002" "claimant:1001"
SCARD claimants:active  # Returns: 2 (duplicate ignored)

Set Operations - Union, Intersection, Difference

Command Description Use Case
SINTER key1 key2 Intersection - common members Find customers with multiple policies
SUNION key1 key2 Union - all unique members Combine customer segments
SDIFF key1 key2 Difference - members in key1 not in key2 Find customers without certain coverage
SINTERSTORE dest key1 key2 Store intersection result Create targeted marketing lists
SUNIONSTORE dest key1 key2 Store union result Combine segments for campaigns
SDIFFSTORE dest key1 key2 Store difference result Identify coverage gaps

Insurance Analytics with Set Operations

# Agent skills and certifications
SADD agent:john:skills "auto" "home" "life" "property_casualty"
SADD agent:jane:skills "auto" "life" "commercial" "umbrella"

# Common skills (intersection)
SINTER agent:john:skills agent:jane:skills
# Returns: ["auto", "life"]

# All unique skills (union)
SUNION agent:john:skills agent:jane:skills
# Returns: ["auto", "home", "life", "property_casualty", "commercial", "umbrella"]

# John's unique skills (difference)
SDIFF agent:john:skills agent:jane:skills
# Returns: ["home", "property_casualty"]

# Find customers with both auto AND home insurance
SADD policy:auto:customers "cust:001" "cust:002" "cust:003"
SADD policy:home:customers "cust:002" "cust:003" "cust:004"

# Multi-policy customers (cross-sell opportunity)
SINTER policy:auto:customers policy:home:customers
# Returns: ["cust:002", "cust:003"]

# Customers with auto but NOT home (upsell target)
SDIFF policy:auto:customers policy:home:customers
# Returns: ["cust:001"]

Redis Sorted Sets - Ranked Collections

What are Sorted Sets?

  • Members associated with scores (floating point)
  • Automatically sorted by score
  • Unique members, scores can duplicate
  • O(log N) insert and lookup

Insurance Applications

  • Claim Rankings: By amount or urgency
  • Agent Performance: Sales leaderboards
  • Customer Scores: Risk ratings, loyalty points
  • Time-Series: Events by timestamp
Key Feature: Maintain automatic ordering by score while ensuring member uniqueness. Perfect for leaderboards, priority queues, and time-based data.

Basic Sorted Set Operations

Command Description Time Complexity
ZADD key score member Add member with score O(log N)
ZSCORE key member Get member's score O(1)
ZINCRBY key increment member Increment member's score O(log N)
ZCARD key Get number of members O(1)
ZREM key member Remove member O(log N)
ZRANK key member Get member rank (0-based, ascending) O(log N)
ZREVRANK key member Get member rank (descending) O(log N)

Sorted Set Range Queries

Command Description Use Case
ZRANGE key start stop [WITHSCORES] Get range by index (ascending) Top N performers
ZREVRANGE key start stop [WITHSCORES] Get range by index (descending) Highest claims first
ZRANGEBYSCORE key min max Get members by score range Claims $10K-$50K
ZREVRANGEBYSCORE key max min Get by score range (descending) High-value policies
ZCOUNT key min max Count members in score range Claims over threshold
Pro Tip: Use WITHSCORES to get both members and their scores in range queries. Essential for displaying claim amounts, scores, or timestamps alongside identifiers.

Claims Ranking by Amount

# Add claims with dollar amounts as scores
ZADD claims:by:amount 2500 "CLAIM-2024-001"
ZADD claims:by:amount 15000 "CLAIM-2024-002"
ZADD claims:by:amount 8500 "CLAIM-2024-003"
ZADD claims:by:amount 125000 "CLAIM-2024-004"
ZADD claims:by:amount 3200 "CLAIM-2024-005"

# Get highest value claims (top 3)
ZREVRANGE claims:by:amount 0 2 WITHSCORES
# Returns: ["CLAIM-2024-004", "125000", "CLAIM-2024-002", "15000", "CLAIM-2024-003", "8500"]

# Get lowest value claims
ZRANGE claims:by:amount 0 2 WITHSCORES
# Returns: ["CLAIM-2024-001", "2500", "CLAIM-2024-005", "3200", "CLAIM-2024-003", "8500"]

# Get specific claim amount
ZSCORE claims:by:amount "CLAIM-2024-004"  # Returns: 125000

# Get claim rank (0-based, lowest to highest)
ZRANK claims:by:amount "CLAIM-2024-004"  # Returns: 4 (highest)
ZREVRANK claims:by:amount "CLAIM-2024-004"  # Returns: 0 (1st place)

# Claims between $5,000 and $20,000
ZRANGEBYSCORE claims:by:amount 5000 20000 WITHSCORES
# Returns: ["CLAIM-2024-003", "8500", "CLAIM-2024-002", "15000"]

# Count high-value claims (over $100K)
ZCOUNT claims:by:amount 100000 +inf  # Returns: 1

# Increment claim amount (revised estimate)
ZINCRBY claims:by:amount 2500 "CLAIM-2024-001"  # Now 5000

Time-Series Data with Sorted Sets

Using Timestamps as Scores

  • Unix timestamps for precise ordering
  • Range queries by time period
  • Automatic chronological sorting
  • TTL cleanup with ZREMRANGEBYSCORE
# Track claim submissions by time
# Score = Unix timestamp
ZADD claims:timeline 1710432000 "CLAIM-001"
ZADD claims:timeline 1710435600 "CLAIM-002"
ZADD claims:timeline 1710439200 "CLAIM-003"

# Get claims from last hour
# (current_time - 3600 to current_time)
ZRANGEBYSCORE claims:timeline \
  1710435600 1710439200

# Remove old claims (older than 30 days)
ZREMRANGEBYSCORE claims:timeline 0 \
  $(($(date +%s) - 2592000))

Sets vs Sorted Sets - When to Use Each

Use Sets When:

  • Order doesn't matter
  • Need set operations (union, intersection, diff)
  • Binary membership testing (in/out)
  • Tagging and categorization
  • Relationship tracking

Example: Customer tags, active claimants, policy features

Use Sorted Sets When:

  • Need ranking or ordering
  • Score-based queries (top N, ranges)
  • Leaderboards and competitions
  • Time-series data
  • Priority queues

Example: Claim rankings, agent leaderboards, risk scores
